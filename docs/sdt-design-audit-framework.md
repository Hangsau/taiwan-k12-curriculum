# SDT 機制級設計稽核框架

> **本檔在專案中的定位（給接手 P4 game design 的 session 看）**
>
> 這是 `docs/motivation-research-collection-mechanics.md`（動機研究）的**操作化續篇**。研究文件給的是「理論 + 原則 + 案例」；本檔把它變成一套**可執行的稽核流程**——拿到任何一個具體遊戲機制，跑完這套檢查，就能輸出 `✅ 可用 / ⚠️ 需 A/B test / ❌ 禁用`，並附理由。
>
> **為什麼需要它**：CLAUDE.md §8 明令「不要把 SDT 三需求寫進 spec 就算交差，每個機制都要對到三需求的具體滿足/受阻狀態」。光讀研究無法落實這句話——設計者拿到「學完一單元抽一張卡」這種具體提案時，需要一個工具判斷它到底行不行。本檔就是那個工具。
>
> **使用前提**：本框架的判定規則，凡是有研究背書的標 `[high]`，凡是從研究推論出來的標 `[推論]`。`[推論]` 的判定上線後要用 A/B test 驗證，不可當作定論。

---

## 0. 怎麼用（30 秒版）

對**每一個**要進遊戲的機制（抽卡、徽章、排行榜、解鎖、進度條、每日任務……），依序跑三道閘門：

```
機制提案
   │
   ▼
Gate 1：框定方式（控制性 vs 資訊性）   ── 控制性 ➜ 紅旗
   │
   ▼
Gate 2：與學習掛鉤（mastery-coupling）  ── 任意累積/脫鉤 ➜ 紅旗
   │
   ▼
Gate 3：SDT 三需求（自主/勝任/關聯）     ── 任一被 thwart ➜ 紅旗
   │
   ▼
判定（§4）
```

- **0 紅旗 + mastery-coupled** → `✅ 可用`
- **唯一的疑慮落在 §3 的「不確定清單」** → `⚠️ 需 A/B test`
- **命中任一 §3「反模式清單」** → `❌ 禁用，回去重設計`

重點：**一個機制只要踩一個 ❌ 反模式，就整個打回**，不能用「其他地方做得好」來抵銷——因為 SDT 三需求任一受阻會產生「質性不同」的成本，不是加總可抵消的（研究 §2.1 `[high]`）。

## 1. 三道閘門（逐題問卷）

把機制丟進來，逐題回答。每個「→ 紅旗」就記一筆。

### Gate 1 — 框定方式：控制性 vs 資訊性 `[high]`

> 同樣一張卡，框定方式不同，效果相反（研究 §2.2.1）。

| # | 問題 | 紅旗條件 |
|---|------|---------|
| 1.1 | 這個機制傳達的訊息是「你做得好、能力提升了」（資訊性），還是「你必須做 / 被要求做」（控制性）？ | 控制性 → 紅旗 |
| 1.2 | 獎勵時機是孩子**可預期且與表現掛鉤**，還是系統強加 / 隨機決定何時給？ | 系統強加或隨機 → 紅旗 |
| 1.3 | 拿掉這個獎勵後，孩子還會想做這件事嗎？（過度合理化測試，研究 §3.1） | 拿掉就不做 → 紅旗 |

### Gate 2 — 與學習掛鉤：mastery-coupling `[high 原理 / 推論 門檻]`

> 蒐集物若能被「內化」成「我學會了」，就走到 OIT 連續線右端，不削弱動機（研究 §2.2.2）。

| # | 問題 | 紅旗條件 |
|---|------|---------|
| 2.1 | 獲得這個蒐集物，能不能**追溯到一個具體學習事件**（學會了哪個概念）？ | 「完成 N 個任務就給」這種與內容脫鉤 → 紅旗 |
| 2.2 | 這個蒐集物有沒有附「它代表你會了什麼」的說明，讓孩子能內化？ | 純裝飾、無學習語意 → 紅旗 |
| 2.3 | 取得條件是**能力**，還是**運氣 / 純累積次數**？ | 運氣或純次數 → 紅旗 |

### Gate 3 — SDT 三需求 `[high 框架 / 推論 個別判定]`

> 三需求任一受阻（thwart）都是獨立成本，不可互相抵銷（研究 §2.1）。

**自主 Autonomy**
| # | 問題 | 紅旗條件 |
|---|------|---------|
| 3.1 | 孩子能不能選擇蒐集目標 / 自訂步調？ | 路徑被單一強制 → 紅旗 |
| 3.2 | 有沒有強制每日登入、streak 中斷懲罰、系統決定「何時該回來」？ | 有任一 → 紅旗 |
| 3.3 | 孩子能不能合理暫停 / 自然結束，而不被懲罰？ | 不能（always-on） → 紅旗 |

**勝任 Competence**
| # | 問題 | 紅旗條件 |
|---|------|---------|
| 3.4 | 回饋是針對**個人挑戰曲線**，還是與他人比較的排名？ | 單一全平台排名 → 紅旗 |
| 3.5 | 落後 / 失敗時，機制是給「下一步怎麼進步」，還是給挫敗訊號？ | 只給挫敗 → 紅旗 |

**關聯 Relatedness**
| # | 問題 | 紅旗條件 |
|---|------|---------|
| 3.6 | 社交元素是**合作 / 分享**，還是排名 / 炫耀稀有度？ | 比較 / 炫耀為主 → 紅旗 |

## 2. 機制 → SDT 速查（給快速判斷用）

研究 §5 的三型蒐集，預先跑過框架的結果：

| 機制型態 | Gate 1 | Gate 2 | Gate 3 | 預設判定 |
|---------|--------|--------|--------|---------|
| **知識/技能解鎖（mastery-based）** | 資訊性 ✓ | 天生 coupled ✓ | 風險最低 ✓ | `✅`（首選）|
| **物件/卡片蒐集（mastery-coupled 版）** | 視框定 | 需設計成 coupled | 看是否有稀有度比較 | `✅` 若 coupled |
| **物件/卡片蒐集（gacha / 隨機抽取版）** | 控制性 ✗ | 運氣脫鉤 ✗ | thwart 自主 ✗ | `❌` |
| **徽章 / 成就（與學習掛鉤）** | 資訊性 ✓ | 需 coupled | 不做公開排名則 OK | `✅`/`⚠️` |
| **排行榜 / streak / 每日登入** | 控制性 ✗ | 常脫鉤 ✗ | thwart 自主+勝任 ✗ | `❌` |

## 3. 紅旗對照清單

### 3.1 ❌ 反模式（命中即 `禁用`）

來自研究 §8.2：

- 隨機抽取 / gacha 機制
- 外顯稀有度顯示（SSR / SR / R）
- 強制每日登入獎勵
- streak 中斷懲罰（移除獎勵或給負面狀態）
- 公開單一全平台 leaderboard
- 任意累積、與學習成就脫鉤的蒐集

### 3.2 ⚠️ 不確定清單（命中則 `需 A/B test`）

來自研究 §8.3，這些缺乏驗證、不能直接放行也不必直接禁：

- 多人連線 / 同儕學習
- 跨領域「集卡」（一個概念學會可拿多領域的卡）
- AI 個人化提示（會不會被當 autonomy-thwarting？）
- 家長可見的進度儀表板（會不會把外在壓力灌進來？）

### 3.3 年齡層加權 `[推論，非已驗證]`

研究 §7 明確標示年齡層差異是 evidence gap。以下只作為**加重保守**的提醒，不是放行/否決依據：

- **低年級（1-3）**：gacha / 外顯稀有度一律當 `❌`；社交炫耀效果強，比較機制要更嚴格。
- **中年級（4-6）**：三型可用，但每個蒐集物都要講清楚「代表什麼能力」。
- **高年級（7-9）**：對「被操弄」敏感，過度外顯蒐集會反噬信任 → 降低蒐集、轉個人成長軌跡 + 同儕協作。

## 4. 判定規則

```
跑完 Gate 1-3，數紅旗：

命中任一 §3.1 反模式            → ❌ 禁用（回去重設計，不可用其他優點抵銷）
否則，疑慮只落在 §3.2 不確定清單  → ⚠️ 需 A/B test 才上線
否則，0 紅旗 且 Gate 2 通過       → ✅ 可用
否則（有紅旗但非反模式）          → ⚠️ 修掉紅旗後重跑，修不掉降 ❌
```

每個判定都要寫一句**理由**，指回觸發的 Gate 題號或研究章節，不能只給結論。

## 5. 範例：用台灣課綱情境跑過一遍

### 範例 A：「學會分數通分 → 解鎖『分數勇者』卡，卡面寫『你會把不同分母變成一樣了』」（數學）
- Gate 1：資訊性（顯示能力提升）✓
- Gate 2：追溯到具體學習事件 ✓、有學習語意說明 ✓、靠能力非運氣 ✓
- Gate 3：自主（自選步調）✓、勝任（個人曲線）✓、關聯（不依賴比較）✓
- **判定：✅ 可用**。理由：mastery-coupled + 資訊性框定，符合研究 §5.3 / §9.2。

### 範例 B：「每日登入抽一張隨機生物卡，集滿圖鑑解鎖自然科主題」（自然科學）
- Gate 1：控制性（強制登入）✗、隨機給 ✗
- Gate 2：運氣取得、與學習脫鉤 ✗
- Gate 3：強制每日登入 thwart 自主 ✗
- **判定：❌ 禁用**。命中 §3.1 兩條反模式（gacha + 強制每日登入）。改法：把「隨機抽」改成「做完一個觀察實驗解鎖對應生物卡」，就轉成範例 A 型態。

### 範例 C：「全班英語單字量排行榜，每週公布前十名」（英語文）
- Gate 1：控制性傾向（比較壓力）✗
- Gate 3：單一公開排名 thwart 勝任（落後者挫敗）✗、比較式社交 thwart 關聯 ✗
- **判定：❌ 禁用**。命中 §3.1 公開 leaderboard。改法：改「個人單字成長軌跡 + 班級合作達標徽章」（合作非比較），重跑可達 `✅`。

### 範例 D：「家長 App 顯示孩子本週學習軌跡儀表板」（跨領域）
- Gate 1：取決於呈現方式
- Gate 3：可能把外在壓力從家長端灌入（autonomy 風險）
- **判定：⚠️ 需 A/B test**。命中 §3.2 不確定清單。設計建議：show「學了什麼」而非「贏了多少 / 排第幾」（研究 §9.2-3）。

### 完整應用範例：寵物 + 居家擺飾遊戲
上面是單一機制的快速判定。一個**完整玩法**怎麼跑過整套框架（含「該拆掉哪些毒核」），見 `docs/p4-design-concept-pet-home.md` — 把電子雞式寵物 + 居家擺飾蒐集逐機制稽核的真實案例。

## 7. 已知限制

- §1 Gate 3 的**個別需求判定**是 SDT 框架的推論性應用 `[推論]`，框架本身（三需求存在）才是 `[high]`。
- §3.3 年齡層加權是 evidence gap（研究 §7），只作保守提醒。
- 本框架判斷「機制設計符不符合 SDT」，**不保證「孩子真的學到 / 長期還會學」**——那是 durability，研究 §6.3 指出對多數案例都沒有通過驗證的答案。框架是「避免踩雷」的工具，不是「保證有效」的保證書。
